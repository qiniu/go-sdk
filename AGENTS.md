# AGENTS.md — Qiniu Go SDK

## Project Overview

Qiniu Cloud Go SDK (`github.com/qiniu/go-sdk/v7`). Go 1.22+, MIT License.

Provides Go clients for Qiniu Cloud services: object storage, CDN, live streaming (Pili), RTC, SMS, IoT (Linking), IAM, media processing, audit, and sandbox.

## Architecture Patterns

### Manager Pattern (storage v1)

The `storage` package uses the Manager pattern:
- `BucketManager` — bucket and object management (stat, copy, move, delete, list)
- `OperationManager` — persistent data processing (pfop)
- `UploadManager` — file uploads

Managers are constructed with `auth.Credentials` and handle HTTP requests and auth internally.

### Provider/Interface Pattern (storagev2)

The `storagev2` package uses interfaces and provider pattern:
- `credentials.CredentialsProvider` — credential provider
- `region.RegionsProvider` — region info provider
- `region.EndpointsProvider` — endpoint provider
- `downloader.DownloadURLsProvider` — download URL provider

### Interceptor Chain (internal/clientv2)

The `internal/clientv2` package implements an HTTP request interceptor chain:
- `Interceptor` interface: `Priority()` + `Intercept(req, handler)`
- Sorted by priority (lower number = higher priority)
- Built-in interceptors: auth, retry, anti-hijacking, header setting, upload logging, debug

### Context-based Authentication

`auth` package supports Context-based credential passing:
- `auth.WithCredentialsType(ctx, cred, tokenType)` — inject credentials into Context
- `auth.CredentialsFromContext(ctx)` — extract credentials from Context

## Code Generation

### Internal API Generator

- Location: `internal/api-generator/`
- Input: YAML specs in `api-specs/`
- Output: Go code in `storagev2/apis/`, `iam/apis/`, `media/apis/`, `audit/apis/`
- Generated files marked: `// THIS FILE IS GENERATED BY api-generator, DO NOT EDIT DIRECTLY!`
- Command: `make generate`

### Sandbox Code Generation

- Control plane API: `oapi-codegen` from OpenAPI specs
- envd HTTP API: `oapi-codegen` from OpenAPI specs
- envd ConnectRPC: `buf` + `protoc-gen-go` + `protoc-gen-connect-go` from Proto
- Command: `make generate-sandbox`

**IMPORTANT**: Never manually edit files marked `DO NOT EDIT`. Modify the API specs and run `make generate` or `make generate-sandbox`.

## Coding Conventions

- Format with `gofmt -s` (enforced by CI)
- Static analysis with `staticcheck` (enforced by CI)
- Comments in Chinese by default
- All exported identifiers must have doc comments (starting with the identifier name)
- Constructor naming: `New*()` or `New*Ex()` (Ex = extended version)
- Root package error type: `QError` (Code + Message) in `err.go`
- Typed errors in `storagev2/errors` package

## Testing

### Build Tags

- Unit tests: `//go:build unit`
- Integration tests: `//go:build integration` (requires environment variables)

### Commands

| Command | Description |
|---------|-------------|
| `make unittest` | Run unit tests |
| `make integrationtest` | Run integration tests |
| `make test` | Run all tests (unit + integration) |
| `make staticcheck` | Run static analysis |

### Conventions

- Use standard `testing` package with `github.com/stretchr/testify/assert`
- Mock by implementing interfaces with custom structs (no mock frameworks)
- Prefer table-driven tests
- Tests exclude `examples` and `sms` directories — do not delete code in these directories

## CI Pipeline

CI triggers on push and pull_request (`.github/workflows/ci-test.yml`):

1. `gofmt -s` format check (stable Go only)
2. `staticcheck` analysis (stable Go only)
3. Build `examples/` (stable Go only)
4. `make unittest` (Go 1.22 + stable)
5. Cross-platform: Linux → Windows → macOS

## Critical Rules

1. **Never edit generated code** — files marked `DO NOT EDIT DIRECTLY` are generated; modify API specs then run `make generate`
2. **Run tests before committing** — `make unittest` and `make staticcheck` must pass
3. **Don't delete sms/examples** — test commands exclude these dirs but their code must remain
4. **Version sync** — when updating version, sync `conf/conf.go`, `README.md`, `CHANGELOG.md`
5. **Submodule** — `api-specs/` is a git submodule; init with `git submodule update --init --recursive`
